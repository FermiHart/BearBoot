# ADR-0002 — Integrity primitive: CRC-64/XZ (ECMA-182)

Status: Accepted

## Context
The protocol promises integrity on every structure. We needed a checksum
that is (a) cheap and dependency-free in a freestanding bootloader and early
kernel, (b) standard and reproducible, (c) strong enough to catch accidental
corruption and casual tampering of small structures. Cryptographic hashes
(SHA-256) are the wrong tool for a non-secret integrity check at this layer —
they need code/space we don't have pre-libc and provide no authenticity
without a key anyway. Authenticity belongs to Secure Boot / measured boot
(the SECURITY tag), not to the framing checksum.

## Decision
Use CRC-64/XZ (ECMA-182): poly 0x42F0E1EBA9EA3693, reflected in/out,
init = xorout = all-ones. Canonical check: CRC64("123456789") =
0x995DC9BBDF1939FA. This is the same checksum xz/Limine use, so existing
tooling validates our structures.

Each INFO/HEADER/TAG carries an 8-byte `checksum` computed over the whole
structure with the `checksum` field itself treated as 8 zero bytes. For a
variable-length tag the CRC spans the full `tag_size` bytes.

A bitwise implementation lives header-only in `bbp_crc64.h` (no 2 KiB table);
a slice-by-8 table variant can be added for bulk hashing if profiling demands.

## Consequences
+ Zero dependencies, runs pre-libc, reproducible, tool-compatible.
+ Detects every single-bit error and burst errors up to 64 bits.
- NOT cryptographic: a deliberate attacker who controls the bytes can forge a
  matching CRC. This is by design — the threat of a malicious *producer* is
  handled by the defensive parser (ADR-0004) bounding damage, and by measured
  boot establishing producer trust, NOT by this checksum. Documented loudly so
  nobody mistakes CRC for authentication.
- Does not cover data referenced by physical pointer outside the struct; see
  ADR-0006.
